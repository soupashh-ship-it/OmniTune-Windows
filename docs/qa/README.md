# OmniTune QA Evidence Policy

This directory contains QA reports, audit notes, runtime verification scripts, and selected evidence for OmniTune Windows.

## Commit deliberately

Commit only evidence that is tied to a specific report, release, or regression investigation.

Good candidates:

- final release-verification reports;
- small targeted screenshots referenced by a report;
- scripted QA evidence that documents a fixed defect;
- manually curated public screenshots under `docs/assets/screenshots/`.

Avoid committing:

- bulk local screenshot sweeps;
- transient runtime logs;
- local download test output;
- generated media files;
- duplicate visual-diff attempts that are not referenced by a report.

## Generated screenshot captures

Local screenshots generated by QA scripts should go under:

```text
docs/qa/screenshots/
```

That folder is ignored by git. If a screenshot is important, copy it into a named report folder and reference it from the report that needs it.

## Runtime artifacts

Runtime download artifacts stay under:

```text
docs/qa/runtime-download-artifacts/
```

That folder is ignored because it can contain generated media and local machine paths.

## Future visual regression direction

The repeatable runner is:

```powershell
.\docs\qa\visual_regression.ps1
```

See:

```text
docs/qa/visual-regression.md
```

The current process is still not a mandatory CI gate. A proper CI visual regression system should:

1. launch a deterministic app state;
2. capture known routes at agreed window sizes;
3. compare against curated baselines with tolerances;
4. store only failed diffs or approved baselines;
5. run as an explicit CI job, not as part of every normal unit-test run.
